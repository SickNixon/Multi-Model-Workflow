## 2. THE PROJECT

You are building **a Tauri-based desktop application for macOS** that acts as a custom multi-panel browser/orchestrator for AI model chat interfaces. The app loads real, live chat websites (Gemini, DeepSeek, Grok, Claude) inside native WebView panels, injects JavaScript to read their outputs and control their inputs, and routes messages between them via a master orchestrator panel — all without using any APIs or API keys.

The goal is a **free, zero-token-cost, multi-model vibe coding workflow tool** where the user can direct prompts, route outputs between models, and optionally loop Claude in for heavier lifting.

- **Tauri 2.x** (Rust backend + WebView frontend shell)
- **Rust** (core application logic, IPC, window management)
- **React + TypeScript** (orchestrator UI panels)
- **JavaScript injection** (DOM reading/writing inside each WebView)
- **macOS WebKit** (WKWebView under the hood via Tauri)
- **Local IPC** (Tauri's command system, events)

**Key data flows to always keep in mind:**
1. User types prompt in Orchestrator Panel → Rust routes it → JS injection writes it into target WebView(s) → monitors for generation complete → reads output → sends back to Rust → Rust routes to next destination

3. The Orchestrator Panel is itself a WebView (React app) but treated differently — it's the control surface, not a model panel
4. Panel state (which models are active, routing rules, conversation history) lives in Rust and is synced to the Orchestrator Panel

## 7. KNOWN CHALLENGES — BE READY FOR THESE

### The Selector Fragility Problem
Every AI chat site updates its UI regularly. Class names change. DOM structure shifts. Your JS injection code will break. Design for this:
- Use semantic/aria selectors where possible (`[role="textbox"]`, `[aria-label="..."]`)
- Build a selector configuration file that can be updated without recompiling
- Implement a "selector health check" that runs on startup and warns the user if something looks broken
- Log every selector attempt with success/failure

### The "Generation Complete" Detection Problem
Knowing when a model has finished generating is non-trivial. Strategies:
- Watch for the send button to re-enable
- Watch for a specific aria-label change
- Use MutationObserver on the output container and debounce when mutations stop
- Implement a configurable timeout as fallback

### The React Controlled Input Problem
Standard `element.value = 'text'` doesn't work on React-controlled inputs. You must:
```javascript
const nativeInputValueSetter = Object.getOwnPropertyDescriptor(
  window.HTMLTextAreaElement.prototype, 'value'
).set;
nativeInputValueSetter.call(element, text);
element.dispatchEvent(new Event('input', { bubbles: true }));
```
For `contenteditable` divs (which Gemini and Claude use), the approach is different — `execCommand` or direct `textContent` manipulation with proper events.

### The WKWebView Content Security Policy Problem
Some sites have strict CSPs. Tauri can override these via configuration, but be explicit about where and why you're doing so. Document every CSP override.

### The Multi-Window State Sync Problem
When multiple WebViews are open and the user switches focus, or a background WebView finishes generating, the Rust core needs clean state management. Use a state machine per-panel, not ad-hoc booleans.
