The goal is a **free, zero-token-cost, multi-model vibe coding workflow tool**.

- **JavaScript injection** (DOM reading/writing inside each WebView)
- **macOS WebKit** (WKWebView under the hood via Tauri)
- **Local IPC** (Tauri's command system, events)

### Target Platform
- macOS (primary)
- Developer machine: Mac, Python 3.14.3, Homebrew, existing Tauri familiarity

## 11. REASONING DISCIPLINE — HOW TO THINK THROUGH HARD PROBLEMS

This project is strictly coding-focused. All reasoning must be efficient, structured, and action-oriented. Unstructured exploration wastes tokens and delays delivery.

**Scope note**: For routine debugging and implementation, follow the structure strictly. For high-level architecture or novel design questions, you may adapt the format slightly while still obeying the Guillotine Rule and anti-patterns.

### Phase 1 — Diagnose Once, Lock In

State the problem in ≤3 sentences. List observed evidence as bullet points — one conclusion per bullet. 

**If the user has already identified the root cause, do not re-diagnose it.** Assume they are correct and jump directly to Phase 2 unless you have clear contradictory evidence. Re-proving user-stated problems is forbidden.

The moment a root cause has ≥2 independent supporting signals, **it is locked**. Stop re-diagnosing. If the same root cause appears a second time in your reasoning, Phase 1 is complete — move on.

### Phase 2 — Options Triage (Fast)

Evaluate candidate solutions using this exact format:
Option A: [short name] — [one-sentence mechanism] — REJECTED: [single reason]
Option B: [short name] — [one-sentence mechanism] — REJECTED: [single reason]
Option C: [short name] — [one-sentence mechanism] — VIABLE: [single reason it survives]
text**Rules**:
- Dead ends get one sentence maximum. Then they are gone forever.
- No zombie options (rejected approaches reappearing reframed).
- If two options are close, note the tradeoff in one sentence and pick one.

### Phase 3 — Commit and Build

**"Committing to [Option X] because [one clear reason]."**

Then implement. Flag specific uncertainties as inline test points. Do not spiral.

**The Guillotine Rule**: If any conclusion, diagnosis, or option has appeared more than once — stop. You are circling. Pick, commit, build.

### Anti-Patterns — Actively Resist These

| Anti-Pattern              | Kill Phrase |
|---------------------------|-------------|
| Re-diagnosis loop         | "Already established. Moving to Phase 2." |
| User-stated re-diagnosis  | "User already identified root cause. Jumping to Phase 2." |
| Zombie options            | "Already rejected. Gone." |
| Hedging paralysis         | "Flag as test point. Committing to most viable." |
| Meta-reasoning            | Delete it. Just reason. |
| Premature implementation  | "Finish triage first." |

### Volume Targets (Guidelines)

- Diagnosis + Triage: **< 600 tokens** total reasoning overhead preferred.
- For genuinely novel/complex problems, override via Section 4.8 (escalate clearly) instead of forcing a weak commit.

### What Good Reasoning Looks Like

**Problem**: IPC output not reaching Rust from Gemini WebView.  
**Evidence**: CSP blocks ipc://, postMessage fallback fails, beacons blocked.  
**Root cause locked**: Communication channel blocked by CSP + external WebView limitations.
Option A: Custom URI scheme — REJECTED: CSP still applies to custom schemes
Option B: Iframe relay — REJECTED: Cross-WebView postMessage not possible
Option C: document.title polling via eval() — VIABLE: Bypasses network/CSP entirely, eval() always works
text**Committing to Option C because** it is reliable and requires no native code changes.
